## 1. What this is

PR Life is a personal life OS. The web app (`~/portfolio`, `/life` + `/api/life`) is the brain. Native companions are thin **capture / display / hardware** layers.

- **iOS app ‚Äî DONE (v1+).** Capture-focused: push-to-talk + lock-screen + on-device transcription ‚Üí POST to `/api/life/entries`. Lives in `~/Developer/PRLifeMobile`.
- **macOS app ‚Äî THIS PROJECT.** A *quiet system utility*: menu-bar popover + main window showing upcoming events/tasks (display-first), desktop-mic quick capture, global keyboard shortcuts, WidgetKit widgets, and a hardware-abstraction seam for future desk-dock/NFC/BLE.
- **Widgets** (iOS+macOS) and **Pebble/Watch hardware** are separate later pieces.

The macOS app is **display-first** (the opposite emphasis from iOS, which is capture-first). It should feel like a calm background utility, not a productivity app. Do NOT duplicate web-app features (tasks/projects/journaling/AI live in the web app).

## 3. Reusable assets (share these ‚Äî don't rewrite)

### `PRLifeKit` SPM package ‚Äî already cross-platform
At repo root (`~/Developer/PRLifeMobile/Package.swift`), `platforms: [.iOS(.v17), .macOS(.v13)]`. Platform-free, 28 passing unit tests (`cd ~/Developer/PRLifeMobile && swift test`). The macOS app should depend on this same package. Modules:

- `Model/` ‚Äî `CaptureStatus`, `CaptureContext` (quick/work/journal/ideas ‚Üí `projectSlug`), `CaptureRecord`, `PRLifeAction` (`.startCapture(context:)`/`.stopCapture`).
- `API/` ‚Äî `EntryPayload`, `LifeAPIClient` (Bearer auth; `upload(content:projectSlug:)`, `deleteEntry(id:)`; supports a lazy `configurationProvider`; throws `.notConfigured`), `Reachability` (`ReachabilityProviding`, `UploadGate` wifi-only logic).
- `Capture/` ‚Äî protocols `CaptureStoring`, `AudioRecording`, `Transcribing`; the `CaptureCoordinator` state machine (start‚Üíprocessing‚Üítranscribe‚Üíuploading‚Üídone/failed, double-start/stop guards, retry-safe `upload`); `CaptureActionRouter` (closure bridge so App Intents in any process reach the live stack); `CaptureControlChannel` (App-Group `UserDefaults` stop-request flag for cross-process stop).
- `Theme/PRLifeTokens.swift` ‚Äî hex color strings (accent `FF3120`, bg `0A0A0A`, etc.) + spacing. **Reuse for macOS.**

### Design system bridge & fonts (copy the pattern)
- `App/Theme/PRLifeTheme.swift` (iOS) ‚Äî `Color(hex:)` + `Theme` enum (Color/Font over `PRLifeTokens`) + `Theme.mono/display/body`. macOS needs its own copy of this bridge (SwiftUI `Color`/`Font` are cross-platform; just re-add it in the macOS target).
- Fonts in `App/Resources/Fonts/`: `ClashDisplay-{Regular,Medium,Semibold,Bold}.otf`, `DMMono-{Light,Regular,Medium}.ttf`. Reuse the same files. Visual language: **square corners, 1px borders, no shadows**, signal red `#FF3120`, DM Mono for labels/meta, Clash Display for headings, SF Pro for body, trailing-underscore labels (`LIFE_`, `CAPTURES_`).

### App Group (already established)
`AppGroup.id = "group.com.pramitranjan.prlife"` (`App/AppGroup.swift`), with entitlements files + SwiftData `ModelConfiguration(groupContainer: .identifier(...))`. The macOS app can share the same group for a shared cache if desired (note: cross-device iOS‚ÜîmacOS does NOT share an App Group container ‚Äî that's per-device; the App Group is for app‚Üîextension on the same device).

## 4. Backend API the macOS app needs (auth: `Authorization: Bearer <LIFE_MOBILE_TOKEN>`)

Auth lives in `~/portfolio/lib/life/auth.ts` (accepts cron secret OR `LIFE_MOBILE_TOKEN`). Set `LIFE_MOBILE_TOKEN` in the deployed env + the app's Keychain config.

- `GET /api/life/calendar` ‚Äî calendars/events source (see `app/api/life/calendar/route.ts`). For the **Upcoming Events** dashboard/widgets.
- `GET /api/life/tasks` ‚Äî tasks (see `app/api/life/tasks/route.ts`). For **Tasks Due Soon**.
- `GET /api/life/entries?date=` , `POST /api/life/entries` ({content, source:"voice", projectSlug}), `DELETE /api/life/entries/[entryId]` ‚Äî capture entries (reused from iOS).
- Other: `app/api/life/{week,month,history,reports,synthesis/*}` exist if needed.

**Verify exact JSON shapes by reading the route files** before modeling ‚Äî don't assume. The macOS dashboard/widgets are the first consumers of `calendar` + `tasks` reads (iOS only used `entries`), so new `PRLifeKit` models (`LifeEvent`, `LifeTask`) + `LifeAPIClient` GET methods + a small cache are NET-NEW work.

## 6. Architecture recommendation for the new chat

- **Where it lives:** add a `PRLifeMac` app target to the SAME `~/Developer/PRLifeMobile` repo + `project.yml` (lowest-friction PRLifeKit reuse; iOS and macOS share one kit and one fonts/theme source). Alternative: a sibling repo path-referencing PRLifeKit ‚Äî more isolation, more wiring. Recommend the monorepo target. Decide in brainstorming.
- **Reuse:** `PRLifeKit` wholesale (models, `LifeAPIClient`, `CaptureCoordinator`, `CaptureActionRouter`, `UploadGate`, `PRLifeTokens`). Re-add the `Theme`/`Color(hex:)` bridge + fonts in the macOS target.
- **Net-new in PRLifeKit:** `LifeEvent`/`LifeTask` models + `LifeAPIClient.fetchEvents()/fetchTasks()` + a lightweight cache/timeline store for the dashboard & widgets (TDD-able, platform-free ‚Äî same pattern as the existing kit).
- **Net-new macOS concretes (app target):** desktop-mic `AudioRecording` impl, a `MenuBarExtra` scene, the Today/Devices windows, a global-hotkey mechanism, optional notifications.
- **Carry-over tech debt to consider:** real-device QA of iOS is still pending (separate); the dead "Background recording" toggle + missing "Audio quality" setting; eventual Swift 6 concurrency migration (the `@MainActor` stores vs `Sendable` protocols are accepted under Swift 5 mode only).
